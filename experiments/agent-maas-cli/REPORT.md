## 4. 逐条评估

### `1.txt`

原文要点：

- 来源为“复旦天协”。
- 活动标题为“路边天文 | 春末夏初，群星交替”。
- 时间为“今晚（5.15）20:00开始”。
- 地点为“光草东北角”。

输出评价：好。

- `source_name`、标题、时间、地点均正确。
- 根据 `reference_date=2026-05-20` 将 `5.15 20:00` 转为 `2026-05-15T20:00:00+08:00`。
- 原文没有结束时间、校区、URL，输出为 `null`，符合抽取规则。
- `evidence_text` 能支持该条活动回溯。

### `2.txt`

- 时间为 `2026.5.10 19:00~21:00`。

### `4.txt`

原文要点：

- 复旦大学 2026 年校园开放日长文。
- 总体地点为复旦大学江湾校区。
- 文中包含多个子活动：开幕式、成果展示、招生咨询、社团嘉年华、草坪音乐节、宿舍参观、校园导览、实验室参观等。

输出评价：中上。

- 长文本不再失败，抽取出 17 条活动，覆盖明显优于旧版。
- 大多数条目都有明确标题、时间、地点和 `evidence_text`。
- `campus=江湾校区` 合理，原文明确写出活动地点为江湾校区。
- 仍存在边界问题：午餐餐券发放、集章抽奖、寄给未来的信等更像配套活动/福利，但也有明确时间地点。是否入库需要产品侧定义“活动候选”的边界。
- “科普实践参观”缺少明确起止时间，输出 `start_time=null`、`end_time=null`，但保留地点和 evidence，适合作为待人工复核候选。

### `5.txt`

原文要点：

- 返校日图书馆活动。
- 主活动为“点亮一本书的生命”线下专场。
- 时间为 `5月23日 12:00-16:00`。
- 地点为“相辉堂草坪和文图”。
- 另有“转发集赞福利”，兑换时间同为 `5月23日 12:00-16:00`，兑换地点为“相辉堂草坪活动摊位”。

输出评价：中等。

- 主活动标题、时间、地点抽取正确。
- 根据 reference date 补全年份为 `2026-05-23`，且原文“周六”与该日期一致。
- 模型把“转发集赞福利兑换”作为第二个活动候选。它有明确时间地点，但更像附属福利，不一定应作为正式 event 入库。
- `warnings` 正确提示文图简称、校区和 organizer 不确定。

## 5. 总体结论

这次重构后，输出结构更贴近数据入库前的抽取任务，不再混用 `plan-day` 运行状态和推荐结果字段。

表现好的部分：

- 新 schema 清晰区分了“抽取事实”和“后续系统字段”。
- JSON 格式稳定，所有输出都通过本地 schema 校验。
- 对短文本、中等文本和长文本都能返回结构化结果。
- 长文本 `4.txt` 已能抽取多条活动，不再被固定条数限制。
- `warnings` 对校区、简称和边界不确定性有帮助。

主要问题：

- “活动”和“福利/配套服务”的边界还不清晰，例如 `5.txt` 的集赞兑换、`4.txt` 的餐券和抽奖。
- `source_name` 对无明确署名的长文仍可能为 `null`，如果上游能传入来源名称会更稳定。
- 长文虽然能抽多条，但仍需要人工判断候选是否应入库。

## 6. 建议

短期 prompt 调整：

- 增加规则：福利、抽奖、集赞、餐券、兑换、纪念品领取默认标记为候选活动，但在 `warnings` 中提示“可能是附属福利”。
- 增加规则：地点字段尽量保留原文简称，不自行扩写。

中期脚本调整：

- 支持上游显式传 `source_name`、`source_url`，减少模型判断来源。
- 增加候选类型字段，例如 `event_type: activity|benefit|visit|consultation|performance|unknown`，帮助后续入库筛选。
- 对长文做分块抽取和合并，降低单次请求上下文压力。
