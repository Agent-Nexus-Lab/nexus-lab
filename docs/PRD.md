# 校园生活 AI 助手 PRD

版本：v0.1  
日期：2026-05-10  
项目阶段：MVP 立项与团队开发拆解  
目标上线窗口：2026-09-01 前完成可演示 Beta

## 1. 项目一句话

面向复旦学生的个性化校园事件 Agent：根据学生显式偏好、历史互动、当前位置/校区、空闲时间和实时更新的校园事件库，推荐可参加、可信、不过期的讲座、会议、社团、运动、演出、志愿和生活活动，并能生成当天或未来几天的个性化安排。

## 2. 背景与机会

大学校园里的活动信息高度分散，常见入口包括复旦官网、院系官网、学生组织公众号、社团群、海报、转发推文和同学口口相传。学生经常面临三个问题：

1. 信息找不到：不知道应该关注哪些公众号、网站和群。
2. 信息不匹配：活动很多，但与个人兴趣、时间、校区和社交偏好不一定匹配。
3. 信息不可信或不新：活动时间、地点、报名状态可能已变更，普通搜索容易给出过期结果。

这个项目的核心不是做一个通用聊天机器人，而是做一个“实时更新的校园事件知识库 + 个性化推荐 Agent”。互联网和公众号是事件入口，系统自己的 Event DB 才是可信检索底座。

## 3. 需求判断

### 3.1 外部需求依据

校园活动与学生参与度、归属感和成长体验直接相关。NSSE 等学生参与度调查长期关注本科生如何参与学校提供的学习与个人发展活动，并将“Learning with Peers”“Campus Environment”等作为重要维度。NACA 也将 campus activities 定义为促进学生连接、参与、发展和归属的课外体验。

这些资料不能直接证明“复旦学生一定会用本产品”，但能支持一个合理判断：校园活动发现和参与本身是高校场景中的真实问题，值得用本校样本继续验证。

### 3.2 本项目需要补做的本地验证

MVP 前两周必须完成小规模需求验证：

1. 访谈 15 到 25 名复旦学生，覆盖大一、大二、研一、不同校区。
2. 发放 100 份以内问卷，重点验证活动发现渠道、错过活动频率、偏好推荐接受度、隐私顾虑。
3. 找 3 到 5 个社团或院系活动运营者访谈，验证他们是否愿意提供活动源或校验入口。

判断继续推进的最低信号：

1. 超过 60% 的受访学生表示“经常或偶尔错过感兴趣的校园活动”。
2. 超过 50% 的受访学生愿意填写偏好表换取推荐。
3. 至少 3 个稳定信息源能被持续采集或人工录入。

## 4. 目标用户

### 4.1 核心用户

1. 大一新生：对校园资源不熟，希望快速找到有趣、低门槛、能认识人的活动。
2. 研一学生：想找讲座、组会、学术沙龙、职业发展活动，但不想每天刷多个公众号。
3. 兴趣明确的学生：例如 AI、游泳、猫、吉他、摄影、辩论、创业、公益等，希望获得跨来源聚合。
4. 轻度迷茫的学生：不知道今天怎么过，需要系统给出“可执行的一天安排”。

### 4.2 次级用户

1. 社团负责人：希望活动被更精准地触达目标学生。
2. 学院/项目运营者：希望讲座、会议、报名通知被学生看到。
3. 产品团队内部运营：需要维护来源、审核低可信事件、查看推荐效果。

## 5. 产品目标与非目标

### 5.0 MVP 范围收缩

为了降低开发压力，MVP 不做完整校园活动流，也不做被动推送。第一版只做一个核心界面：

```text
日程页 plan-day 输入 -> 活动检索 -> 个性化排序 -> 生成今日/未来安排 -> 展示候选活动卡片
```

MVP 保留：

1. 微信小程序日程页。
2. 日程页内 plan-day 输入入口。
3. 简化偏好表。
4. 活动卡片。
5. 日程生成。
6. 来源与验证状态。

MVP 暂缓：

1. 首页推荐流。
2. 被动推送和订阅提醒。
3. 独立搜索页。
4. 完整“我的收藏”体系。
5. 社交搭子、活动组织者后台和复杂运营看板。

### 5.1 MVP 目标

1. 用户能填写偏好表，形成可编辑的兴趣画像。
2. 系统能持续收集和结构化校园活动，形成 Event DB。
3. 用户输入日程规划需求后，系统能返回个性化、时间敏感、带来源证据的安排。
4. 用户能让 Agent 生成“今天/明天/本周末安排”。
5. 系统能对高风险字段进行验证：时间、地点、报名状态、来源可信度。
6. 团队能通过后台查看事件、来源、失败采集任务和用户反馈。

### 5.2 非目标

1. 不做完整校园门户，不替代学校官网和社团平台。
2. 不在 MVP 阶段做全自动抢票、自动报名或代填表。
3. 不承诺覆盖所有校园活动，先覆盖高质量来源和热门场景。
4. 不在 MVP 阶段自训练大模型；先用 API 和开源 embedding/reranker 组合。
5. 不把 MiroThinker 等研究 Agent 直接作为所有请求的主模型；MVP 先实现自己的受控 Runtime。

## 6. 核心场景

### 场景 A：主动搜索

用户输入：“今晚江湾附近有什么 AI 相关活动？”

系统返回：

1. 3 到 5 个活动卡片。
2. 每个卡片包含时间、地点、校区、活动类型、推荐理由、来源链接、可信度、报名状态。
3. 明确说明是否已验证，以及验证时间。

### 场景 B：一天安排

用户输入：“我今天不想浪费掉，帮我安排一下。”

系统结合偏好、空闲时间、校区和活动库，生成：

1. 上午/下午/晚上分段安排。
2. 学术、运动、兴趣、轻社交的平衡。
3. 每项活动的备选方案。
4. 通勤和时间冲突提醒。

### 场景 C：模糊兴趣探索

用户输入：“有没有适合 introvert 的活动？”

系统不机械搜索 introvert，而是推理为“小型、低压、非强社交、可旁听、工作坊、读书会”等标签，返回匹配活动。

### 场景 D：活动订阅（P1/P2，MVP 暂不做）

用户设置兴趣：“AI、游泳、猫、吉他、江湾、晚上 18:00 后。”

系统在每天固定时间推送或展示：

1. 今日推荐。
2. 未来 3 天值得关注。
3. 新增活动提醒。
4. 报名截止提醒。

MVP 决策：

1. 暂不做被动推送。
2. 暂不做每天固定时间主动推荐。
3. 只保留用户主动进入“日程”界面后的搜索与规划入口。

### 场景 E：反馈闭环

用户对推荐点击“不感兴趣”“时间不合适”“已过期”“地点错了”。

系统记录反馈，用于：

1. 调整用户画像。
2. 降低事件或来源可信度。
3. 进入人工审核队列。

## 7. 功能需求

### 7.1 用户自由偏好描述

MVP 不强制用户选择标签，避免用户产生“被定义”的感受。前端采用自由输入为主，结构化字段由后端从用户文本中抽取。

用户看到的是开放问题：

1. 你平时在哪个校区活动比较多？
2. 你现在大概是什么身份？例如本科、研究生、老师等。
3. 你最近对什么事情感兴趣？
4. 你通常什么时候有空？
5. 你更喜欢什么样的活动氛围？

后端抽取为结构化画像：

1. 校区：邯郸、江湾、枫林、张江、其他。
2. 身份：本科、硕士、博士、教职工、其他。
3. 兴趣标签：学术、AI、计算机、金融、艺术、音乐、运动、公益、职业发展等。
4. 可活动时间：工作日白天、工作日晚间、周末。
5. 活动风格：安静旁听、小型交流、大型讲座、强社交、户外、运动。
6. 常用地点、不喜欢的活动类型、预算范围、同伴偏好等选填信息。

验收标准：

1. 3 分钟内可完成。
2. 用户可随时修改。
3. 不填写也能获得推荐。
4. 系统展示“我理解到的偏好摘要”，允许用户修改，不强迫用户选择固定标签。

### 7.2 活动卡片

字段：

1. 标题。
2. 时间和持续时长。
3. 地点、校区、线上/线下。
4. 活动类型和标签。
5. 来源名称和原文链接。
6. 报名方式和截止时间。
7. 推荐理由。
8. 可信度与验证状态。
9. 收藏、反馈、分享。

验收标准：

1. 每个推荐必须有来源。
2. 已过期活动默认不展示，除非用户明确查历史。
3. 时间不确定时必须标注“不确定”，不能编造。

### 7.3 自然语言搜索

支持 query 类型：

1. 主题搜索：“最近有什么 AI 讲座？”
2. 时间搜索：“今晚有什么活动？”
3. 地点搜索：“江湾附近有什么？”
4. 风格搜索：“适合一个人去的活动。”
5. 规划搜索：“帮我安排周六下午。”

验收标准：

1. P95 响应时间小于 8 秒；需要实时验证时允许小于 20 秒。
2. 至少返回 1 个“为什么推荐”的解释。
3. 没有结果时给出扩大条件建议，而不是胡编。

### 7.4 今日/未来安排生成

输入：

1. 用户 query。
2. 用户画像。
3. 时间范围。
4. 可选地点。
5. Event DB 候选活动。

输出：

1. 分时段日程。
2. 交通/换场提醒。
3. 冲突检查。
4. 备选活动。

验收标准：

1. 同一时间不能安排两个线下活动。
2. 活动开始前预留合理通勤时间。
3. 每个安排项都能回溯到事件卡片。

### 7.5 事件采集与管理

MVP 来源优先级：

1. 复旦官网、院系官网、通知公告。
2. 可信学生组织公众号或网页。
3. 社团活动表单、飞书/腾讯文档等经授权来源。
4. 人工录入和用户投稿。

后台能力：

1. 来源列表管理。
2. 采集任务状态。
3. 事件审核。
4. 重复事件合并。
5. 低可信事件标记。

验收标准：

1. 每天自动更新至少 2 次。
2. 采集失败可追踪。
3. 人工可修正标题、时间、地点、链接、标签。

### 7.6 个性化与记忆

记忆类型：

1. 显式画像：用户填表。
2. 行为反馈：收藏、点击、不感兴趣、纠错。
3. 对话偏好：用户在聊天中明确说出的长期兴趣。
4. 会话摘要：当前会话中压缩出的临时上下文。
5. 推断偏好：从多次行为中归纳出的稳定倾向，但需标注置信度。

会话要求：

1. 用户侧只看到连续对话，不需要理解底层 session 切换。
2. 系统必须自动管理 session 的创建、续期、摘要和截断。
3. 长对话必须在不丢失关键偏好的前提下压缩上下文。
4. 对话线程和长期记忆必须可区分，不能混在一起。

隐私要求：

1. 用户可查看和删除自己的画像。
2. 不采集敏感个人信息，除非功能明确需要且用户同意。
3. 不把用户个人画像暴露给活动组织者。
4. 记忆项需要可解释、可过期、可删除、可审计。
5. 前端暂定仅做微信小程序，session 和上下文管理由服务端完成。

## 8. 推荐与排序规则

活动排序采用加权分：

```text
score =
  semantic_match * 0.25
+ interest_match * 0.25
+ time_fit * 0.20
+ location_fit * 0.10
+ source_reliability * 0.10
+ freshness * 0.05
+ feedback_adjustment * 0.05
```

硬过滤：

1. 活动结束时间早于当前时间。
2. 地点缺失且无法验证，除非是线上活动。
3. 来源可信度过低且未审核。
4. 与用户明确排除偏好冲突。

## 9. 成功指标

### 9.1 MVP 指标

1. 事件覆盖：Beta 前结构化有效事件累计 500 条以上。
2. 新鲜度：高优先级来源 24 小时内更新。
3. 推荐可用性：人工评测 Top5 推荐中至少 3 条相关。
4. 时间准确率：抽样活动时间/地点准确率 90% 以上。
5. 用户留存：内测用户次周留存 25% 以上。
6. 反馈率：推荐卡片反馈率 5% 以上。

### 9.2 体验指标

1. 首次偏好表完成率 70% 以上。
2. 搜索无结果率低于 20%。
3. P95 普通搜索响应小于 8 秒。
4. P95 深度规划响应小于 20 秒。

## 11. 技术路线概览

MVP 推荐技术栈：

1. 前端：微信小程序日程页优先，必要时再补管理后台 Web。
2. 后端：Python FastAPI。
3. 数据库：PostgreSQL + pgvector。
4. 缓存/队列：Redis + Celery/RQ。
5. 检索：Postgres full-text/BM25 方案 + pgvector embedding。
6. Rerank：bge-reranker 或轻量 cross-encoder。
7. LLM：DeepSeek/Qwen/OpenAI 任选一到两个供应商做 adapter。
8. 部署：Docker Compose 起步，后续再拆服务。

MiroThinker 类模型的定位：

1. MVP 不作为主模型。
2. P1/P2 可作为复杂搜索 Planner 的实验选项。
3. 学习它的行为模式：Think -> Search -> Verify -> Reflect -> Re-plan。
4. 不学习无限工具调用，MVP 工具调用预算控制在 5 次以内。

## 12. 成本与设备预估

以下价格按 2026-05-10 可查公开价格做估算，实际以供应商账单为准。

### 12.1 模型 API

参考价格：

1. OpenAI gpt-5.4-mini：输入约 $0.375/百万 tokens，输出约 $2.25/百万 tokens。
2. DeepSeek v4-flash：输入 cache miss 约 $0.14/百万 tokens，输出约 $0.28/百万 tokens。
3. 阿里云百炼 qwen-plus：中国内地 128K 内输入约 0.8 元/百万 tokens，非思考输出约 2 元/百万 tokens，思考输出约 8 元/百万 tokens。

内测成本假设：

1. 50 DAU，每人每天 5 次请求，约 7,500 次/月。
2. 70% 普通搜索，30% 深度规划。
3. 普通请求平均 2,000 输入 tokens + 800 输出 tokens。
4. 深度请求平均 8,000 输入 tokens + 1,500 输出 tokens。

粗略月成本：

1. 全部使用 DeepSeek/Qwen 轻量模型：约 20 到 100 元/月量级。
2. 普通请求用 DeepSeek/Qwen，深度规划用 gpt-5.4-mini：约 100 到 300 元/月量级。
3. 若大量使用付费搜索 API、网页抽取 API，可能额外 50 到 500 元/月。

### 12.2 服务器

MVP 内测：

1. 2C4G 云服务器 + 80G 磁盘：约 50 到 120 元/月。
2. Beta 推荐 4C8G + 100G 磁盘：约 150 到 350 元/月。
3. 对象存储和日志：早期约 0 到 50 元/月。
4. 域名和备案：域名约 50 到 100 元/年；备案按云厂商流程。

不建议 MVP 自建 GPU 常驻推理。若实验开源模型，可短时租 24GB/48GB GPU 做测试；长期常驻 GPU 往往会把月成本推到数千元以上，不适合学生团队初期。

## 14. 风险与决策点

### 14.1 数据源风险

风险：公众号和网页格式不稳定，部分来源无法合规自动采集。  
策略：先用可授权、可订阅、可人工维护的来源；高价值来源可以半自动录入。

### 14.2 推荐质量风险

风险：活动少时个性化效果不明显。  
策略：先覆盖高频主题和高活跃来源；没有足够结果时扩大时间和主题，并明确说明。

### 14.3 幻觉风险

风险：Agent 编造活动、地点、报名链接。  
策略：所有活动必须来自 Event DB 或验证来源；没有证据不展示为事实。

### 14.4 团队经验风险

风险：4 名大一成员工程经验不足。  
策略：降低技术复杂度，采用单体后端 + Docker Compose；每周 code review；接口先行。

### 14.5 成本风险

风险：深度搜索和大模型调用失控。  
策略：Router 分流；普通请求走便宜模型；深度规划限制工具调用；缓存检索结果。

## 15. MVP 验收清单

上线前必须满足：

1. 至少 500 条结构化有效事件。
2. 至少 10 个稳定来源或人工维护来源。
3. 用户可完成偏好表、搜索、收藏、反馈。
4. 搜索结果必须带来源链接。
5. 今日安排不能出现明显时间冲突。
6. 后台能修正错误事件。
7. 有成本监控和错误日志。
8. 有隐私说明和删除画像入口。
