## 二、当前问题、原因与改进方向

### 1. Memory 系统当前问题

负责人标记：

```text
曹昕宇：主责后端 DB/API/read_memory，负责 feedback 保存、memory 聚合、PlanRun.debug 对接。
马梓腾：配合定义 search_events/scoring 能消费的 Memory 字段，保证 memory 能进入排序。
李颖哲：配合定义用户反馈语义和排除语义，保证 intent 能区分硬约束、软偏好和反馈表达。
张子恒：主责前端反馈入口，保证结果页能把 event_id、plan_id/plan_item_id、run_id、feedback_type、feedback_source、metadata 回传后端。
```

#### 问题 1：memory 目前还没有完整闭环

当前代码中 runtime 已经有部分 memory 文案支持，例如 `build_summary` 和 `build_reason_text` 可以接收 memory 参数。但整体链路还没有形成：

```text
用户反馈
-> 保存 feedback
-> 生成或更新 memory
-> read_memory 读取
-> search_events 使用 memory 调整排序
-> 前端展示 memory 影响
```

目前缺口包括：

```text
1. 缺 user_event_feedback 表。
2. 缺 memory_items 表。
3. 缺 memory_audit_log 表。
4. 缺 POST /api/feedback/event。
5. 缺 read_memory 聚合服务。
6. 正式 plan-day 尚未稳定把 memory 传入 search_events。
7. 前端结果页还缺完整反馈动作与字段回传。
```

#### 原因

第一阶段 MVP 主要验证 `user request -> return_result` 能否跑通，memory 当时只是预留位置。第二阶段要让系统从“一次性生成”变成“能基于历史和反馈调整推荐”，所以需要把 feedback、memory、scoring 三块接起来。

#### 改进方式

本周期只做轻量 memory，不做复杂向量记忆。推荐最小闭环：

```text
user_event_feedback：保存用户行为
memory_items：保存可复用偏好
read_memory：把反馈/历史聚合为 Memory 对象
search_events：根据 Memory 做软加权/降权
debug：返回 memory 对排序的影响
```

`Memory` 建议先支持这些字段：

```text
recent_query_texts
recent_plan_event_ids
liked_tags
disliked_tags
negative_keywords
liked_event_ids
disliked_event_ids
```

更具体的落地要求：

```text
1. user_event_feedback 只记录用户对某个活动的真实动作，不直接等同于长期偏好。
   建议字段：event_id、profile_id、feedback_type、feedback_source、comment、created_at、plan_id、plan_item_id、run_id、metadata。
   feedback_type 先支持 like / dislike / clicked_source。

2. memory_items 保存从多次反馈中沉淀出的轻量偏好。
   建议字段：profile_id、memory_type、key、value、weight、confidence、source、updated_at。
   memory_type 先支持 liked_tag、disliked_tag、negative_keyword、liked_event、disliked_event。

3. read_memory 不直接拼 prompt，而是返回结构化 Memory 对象。
   search_events/scoring 读取这个对象后调整分数。
   rewrite summary/reason 可以读取这个对象生成解释文案。

4. memory 更新要有来源。
   如果某个 disliked_tag 来自用户连续 2 次 dislike 同类活动，需要能在 debug 或 audit 中看到来源。
   不能凭一次偶然点击就把用户永久定义成“不喜欢某类活动”。

5. memory 必须有衰减或覆盖机制。
   本周可以先简单做：最近 30 天有效，或者每次更新 updated_at，后续再做更复杂的时间衰减。

6. 空 memory 不能影响主链路。
   如果用户没有历史反馈，read_memory 应返回空结构，search_events 分数应与原逻辑保持一致。
```

### 2. Memory scoring 当前问题

负责人标记：

```text
马梓腾：主责 search_events/scoring.py，完成 memory 软加权/降权、score_components 和测试覆盖。
曹昕宇：配合提供 read_memory 输出结构，并保证 plan-day 调用 search_events 时能传入 memory。
李颖哲：配合确认 intent 中哪些是 hard_constraints，哪些只是 soft_constraints/negative_keywords。
张子恒：配合调试前端 debug 展示，不要求完整展示所有 score_components，但开发模式要能看见关键信息。
```

#### 问题

`origin/feature/agent-core-mvp` 中已经有一版 memory scoring 思路：

```text
recent_plan_event_ids 命中：repeat_penalty = -0.15
disliked_tags 命中：每个 -0.10，最多 -0.20
negative_keywords 命中：每个 -0.10，最多 -0.20
liked_tags 命中：每个 +0.10，最多 +0.20
```

这个方向合理，因为它是软调整，不直接过滤活动。但仍有两个风险：

```text
1. disliked_tags 和 negative_keywords 可能重复扣分。
2. liked_tags 和 interest_match 可能重复加分。
```

例如用户不喜欢“毕业季”，某活动既有 tag `毕业季`，标题里也有“毕业季”，可能同时触发 tag penalty 和 keyword penalty，导致惩罚过强。

#### 改进方式

本周期 scoring 改进要求需要写到可实现、可解释、可测试的程度。

##### 2.1 memory 只做软加权/降权，不直接 reject

含义：

```text
memory 只能改变 score，不能把活动从 candidates 中删除。
```

正确做法：

```text
如果用户历史上 dislike 过“创业讲座”，这次检索到创业讲座：
保留该活动
score 减少
debug 写明：因为历史反馈命中 disliked_tags，所以降权
```

错误做法：

```text
if event.tag in memory.disliked_tags:
    continue
```

原因：

```text
用户历史偏好不是当前明确约束。
用户以前不喜欢某类活动，不代表这次一定不想看。
如果 memory 直接 reject，会导致推荐越来越窄，用户很难跳出旧偏好。
```

只有当前 query 中明确出现的硬约束才允许 reject：

```text
用户说“不要周五”：周五活动可以 reject。
用户说“只看今天下午”：不在今天下午的活动可以 reject。
用户历史上较少去北校区：北校区活动不能 reject，只能轻微降权。
```

##### 2.2 所有 memory 影响必须写入 score_components

每个候选活动最终返回 debug 时，至少能看到：

```json
{
  "event_id": "evt_123",
  "base_score": 0.62,
  "score_components": {
    "intent_match": 0.28,
    "time_match": 0.14,
    "campus_match": 0.05,
    "freshness": 0.08,
    "memory": {
      "liked_boost": 0.10,
      "disliked_penalty": 0.00,
      "keyword_penalty": -0.10,
      "repeat_penalty": 0.00,
      "total_memory_delta": 0.00,
      "matched_memory_terms": ["展览", "排除:讲座"],
      "explanation": "命中喜欢标签“展览”，同时标题命中负向关键词“讲座”，正负影响抵消"
    }
  },
  "final_score": 0.62
}
```

最低要求：

```text
liked_boost：喜欢标签、喜欢活动相似项带来的加分
disliked_penalty：不喜欢标签、不喜欢活动相似项带来的降分
keyword_penalty：负向关键词命中标题、地点、正文摘要带来的降分
repeat_penalty：近期已推荐过同一活动带来的降分
total_memory_delta：上述 memory 分数合计，经过上下限裁剪后的最终值
matched_memory_terms：具体命中的标签、关键词、event_id
```

##### 2.3 每一种 penalty/boost 必须可解释

不能只返回一个数字：

```json
{ "memory_score": -0.2 }
```

需要能回答三个问题：

```text
为什么加分或扣分？
命中了哪个 memory？
这条 memory 来自哪里？
```

建议结构：

```json
{
  "type": "disliked_penalty",
  "delta": -0.10,
  "matched": "创业",
  "matched_field": "tags",
  "source": "user_event_feedback",
  "source_event_id": "evt_001",
  "reason": "用户曾对包含“创业”标签的活动点过不感兴趣"
}
```

前端不一定全部展示，但后端 debug 必须能看见，方便联调和验收。

##### 2.4 重复扣分和重复加分要做上限控制

建议规则：

```text
liked_boost 最大 +0.20
disliked_penalty 最大 -0.20
keyword_penalty 最大 -0.20
repeat_penalty 单独最大 -0.15
单个活动 total_memory_delta 限制在 [-0.30, +0.20]
```

例子：

```text
某活动同时命中：
liked_tags: +0.10
disliked_tags: -0.20
negative_keywords: -0.20
recent_plan_event_ids: -0.15

原始 memory_delta = -0.45
裁剪后 total_memory_delta = -0.30
```

原因：

```text
memory 是辅助排序，不是裁判。
如果扣分无限叠加，一个活动可能因为标题和 tag 重复命中被打到完全没有机会。
如果加分无限叠加，系统也会过度迎合旧偏好。
```

##### 2.5 liked/disliked 同时命中的处理规则

同一个活动可能同时命中正向和负向 memory，例如：

```text
用户喜欢“展览”
用户不喜欢“讲座”
活动标题：“艺术展览与学术讲座”
```

处理要求：

```text
1. 正向和负向都保留，不要互相覆盖。
2. score_components 中分别写 liked_boost 和 keyword_penalty/disliked_penalty。
3. final_score 使用合计后的 total_memory_delta。
4. explanation 写明“既命中喜欢项，也命中排除项”。
```

不能写成：

```text
只要 disliked 命中，就把 liked_boost 清零。
只要 liked 命中，就忽略 disliked_penalty。
```

##### 2.6 必须补的测试 case

至少补 5 组单元测试或最小集成测试：

```text
Case 1：空 memory
输入：memory 所有字段为空。
预期：final_score 与原始 scoring 一致，score_components.memory.total_memory_delta = 0。

Case 2：liked_tags 命中
输入：liked_tags = ["展览"]，活动 tags/title 命中“展览”。
预期：liked_boost > 0，final_score 上升，不 reject。

Case 3：disliked_tags 和 negative_keywords 同时命中
输入：disliked_tags = ["创业"]，negative_keywords = ["商业路演"]。
预期：disliked_penalty 和 keyword_penalty 都有记录，但 total_memory_delta 不低于 -0.30。

Case 4：近期重复推荐
输入：recent_plan_event_ids 包含当前 event_id。
预期：repeat_penalty < 0，活动仍保留在候选中。

Case 5：liked/disliked 同时命中
输入：liked_tags = ["展览"]，disliked_tags = ["讲座"]，活动同时命中。
预期：liked_boost 和 disliked_penalty 同时存在，debug 可解释最终分数。
```

验收时至少要能在 debug 中看到一个活动因为 memory 被加分、一个活动因为 memory 被降分、一个活动虽然被降分但没有被 reject。

### 3. Loading 时间过长问题

负责人标记：

```text
曹昕宇：主责后端 timings、cache、rewrite 开关、fallback、stage/progress/stage_message 字段。
李颖哲：主责 LLM rewrite prompt、prompt_version、JSON 稳定性、rewrite cache 版本字段和 fallback 文案。
张子恒：主责 loading 页阶段展示、cache_hit 文案、failed/debug 展示。
马梓腾：配合输出 search_events/filter_and_score 耗时和候选快照信息，避免 cache key 与排序结果脱节。
```

#### 问题

目前前端已经是：

```text
POST /api/agent/plan-day
-> loading 页轮询 GET /api/agent/runs/{run_id}
-> completed 后跳结果页
```

但用户体感仍然慢。

#### 原因判断

当前硬编码 intent parser 不是主要瓶颈。`intent_parser.py` 默认 `use_llm=False`，本地规则解析非常快。真正可疑点是：

```text
1. 后端每次 plan-day 都 enable_llm_rewrite=True。
2. LLM rewrite 需要远程 MaaS 请求。
3. LLM_TIMEOUT 当前可到 30s。
4. llm.py 中还有重试机制。
5. POST /api/agent/plan-day 仍偏同步执行，容易让用户感觉一直在等。
```

#### 改进方式

本周期先做可控优化，核心是先证明慢在哪里，再缓存最贵的部分。

##### 3.1 增加阶段耗时 timings

每次 plan-day 至少记录这些耗时：

```json
{
  "timings_ms": {
    "parse_intent": 12,
    "load_profile": 8,
    "read_memory": 15,
    "search_events": 43,
    "filter_and_score": 20,
    "build_schedule": 18,
    "llm_rewrite_summary_and_reasons": 6200,
    "save_plan": 35,
    "total": 6421
  }
}
```

要求：

```text
1. timings 写入 PlanRun.debug。
2. 开发模式 GET /api/agent/runs/{run_id} 能返回 timings。
3. 不再凭感觉判断 intent/parser 是否慢，要用 timings 证明。
4. 如果 LLM rewrite 超过 3 秒，debug 中要能看出来。
```

##### 3.2 24h cache 优先缓存高成本结果

本周优先做两类缓存：

```text
rewrite_cache：缓存 LLM summary/reason 改写结果。
plan_result_cache：缓存最终 plan result。
```

rewrite_cache 建议 key：

```text
rewrite:{profile_id}:{query_hash}:{candidate_ids_hash}:{prompt_version}:{model_name}
```

plan_result_cache 建议 key：

```text
plan_result:{profile_id}:{query_hash}:{profile_hash}:{memory_hash}:{event_snapshot_hash}:{runtime_version}
```

字段要求：

```text
cache_hit: true/false
cache_type: none / rewrite / plan_result
cache_key: 开发模式可返回，生产模式可隐藏或脱敏
cache_ttl_seconds: 剩余有效期
cache_created_at: 首次写入时间
cache_version: 当前缓存版本
```

##### 3.3 cache 命中后不能绕过必要检查

命中 plan_result_cache 后仍要检查：

```text
1. profile_id 是否一致。
2. plan 对应活动是否仍存在。
3. 活动是否已经过期。
4. 活动是否被标记取消或不可用。
5. 当前 memory_hash 是否变化。
```

如果 memory 已变化，不能直接复用旧 plan_result_cache，可以复用 rewrite_cache 或重新排序。

##### 3.4 enable_llm_rewrite 要做成配置项和 fallback

要求：

```text
1. 开发环境可以通过环境变量关闭 LLM rewrite。
2. LLM rewrite 超时不能导致整个 plan-day failed。
3. rewrite 失败时使用模板 summary/reason fallback。
4. debug 写明 rewrite_failed、rewrite_error、used_fallback。
```

建议字段：

```json
{
  "llm_rewrite": {
    "enabled": true,
    "cache_hit": false,
    "timeout_ms": 30000,
    "used_fallback": false,
    "error": null
  }
}
```

##### 3.5 Loading 页面展示要和后端阶段对齐

后端 GET /api/agent/runs/{run_id} 建议返回：

```json
{
  "status": "running",
  "stage": "search_events",
  "progress": 45,
  "stage_message": "正在检索可参加的校园活动",
  "cache_hit": false
}
```

阶段枚举建议：

```text
intent_parsing：正在理解你的需求
load_profile：正在读取你的画像
read_memory：正在读取你的偏好记忆
search_events：正在检索可参加的活动
filter_and_score：正在根据时间、地点和偏好排序
build_schedule：正在编排日程
rewrite_plan：正在整理推荐理由
save_plan：正在保存规划结果
cache_hit：命中缓存，正在快速返回结果
```

前端要求：

```text
1. 如果后端返回 stage_message，优先展示后端文案。
2. 如果后端没返回 stage_message，根据 stage 映射默认文案。
3. 如果 cache_hit=true，展示“命中缓存，正在返回上次可复用结果”。
4. failed 状态展示 debug/rejection reason，方便联调。
```

### 4. 流式生成问题

负责人标记：

```text
张子恒：主责前端流式可行性验证，确认 wx.request enableChunked 或替代方案是否稳定。
曹昕宇：配合提供 dev-only stream/chunk 测试接口或后端阶段字段，不影响正式 plan-day 主链路。
李颖哲：配合确认 rewrite 输出是否适合逐段展示，但本周不把 rewrite 流式化作为强制目标。
马梓腾：配合确认流式探索不影响 search_events/scoring 主线。
```

#### 判断

真正的流式生成方向正确，但本周期不建议作为强制交付。微信小程序里完整流式涉及：

```text
1. 新增 stream endpoint。
2. 后端分块返回 stage/token/result。
3. 前端使用 wx.request enableChunked 或其他兼容方案。
4. 处理失败、重连、最终结果落库。
```

工程量和兼容风险较高。

#### 改进方式

本周期采用两层策略：

```text
主线：阶段式 loading，继续使用现有轮询。
探索线：前端验证 wx.request enableChunked 是否可用，做最小 demo，不影响主链路。
```

阶段式 loading 的目标是先让用户看到：

```text
正在理解需求
正在读取记忆
正在检索活动
正在编排日程
正在整理推荐理由
命中缓存，正在直接返回结果
```

更具体的执行要求：

```text
1. 正式主链路继续使用 POST + GET 轮询，不在本周强行切换为 stream。
2. 流式 demo 可以单独建临时页面或 dev-only endpoint，不影响 plan-day 正式接口。
3. demo 只需要证明小程序端能收到分块数据，不要求接完整 agent runtime。
4. 如果 enableChunked 在当前基础库或真机环境不稳定，需要记录限制，不要影响本周验收。
5. 周五验收以“阶段式 loading + cache 命中”为主，流式结果只作为技术调研结论。
```

流式探索需要输出一段结论：

```text
当前微信小程序环境是否支持稳定 chunk 接收。
需要后端提供 SSE、chunked HTTP，还是 WebSocket。
如果进入第三阶段正式开发，需要改哪些接口。
如果不进入第三阶段，阶段式 loading 是否足够支撑结项演示。
```

## 五、各模块改进要求总表

### 1. Memory / Scoring

```text
问题：
当前 memory scoring 如果只写“加权/降权”，大家容易理解成随便加一个分数。
真正需要的是：每一次加分、扣分都能解释、能复现、能在 debug 中看到。

原因：
memory 是产品个性化能力的起点。
如果 memory 影响不可解释，后面用户问“为什么推荐这个”“为什么不推荐那个”时无法定位。
如果 memory 影响过强，会导致推荐被历史行为锁死。

怎么改：
1. search_events 接收结构化 memory，不接收散乱字符串。
2. scoring.py 中单独计算 liked_boost、disliked_penalty、keyword_penalty、repeat_penalty。
3. 每个分数组件都写入 score_components.memory。
4. total_memory_delta 做上下限裁剪，建议 [-0.30, +0.20]。
5. memory 命中只影响排序，不直接 reject candidates。
6. liked/disliked 同时命中时，两个分量都保留，最后合并。
7. recent_plan_event_ids 只做 repeat_penalty，避免短时间重复推荐。

验收证据：
1. debug 中能看到 score_components.memory。
2. 至少一条活动有 liked_boost。
3. 至少一条活动有 disliked_penalty 或 keyword_penalty。
4. 至少一条活动命中 repeat_penalty 后仍保留在候选中。
5. 单测覆盖空 memory、liked、disliked、重复推荐、正负同时命中。
```

### 2. Feedback / Read Memory

```text
问题：
前端点击反馈如果只存在页面状态里，下一次推荐无法使用。
后端如果只保存 feedback，不聚合成 memory，search_events 也无法消费。

原因：
feedback 是一次用户行为，memory 是可复用偏好。
二者需要分开存储、分开解释。

怎么改：
1. POST /api/feedback/event 保存用户行为。
2. user_event_feedback 存 event_id、profile_id、feedback_type、feedback_source、comment、plan_id、plan_item_id、run_id、metadata、created_at。
3. read_memory 读取最近反馈和最近推荐历史，聚合成 liked_tags、disliked_tags、negative_keywords、recent_plan_event_ids。
4. memory_items 可以先做轻量表，也可以先用聚合查询临时生成，但输出结构要稳定。
5. 对 dislike 不要立刻永久化成强偏好，至少要保留 source 和 confidence。

验收证据：
1. 点击“不感兴趣”后，数据库或接口能查到 feedback。
2. read_memory 返回包含 disliked_event_ids 或 disliked_tags。
3. 下一次 plan-day debug 显示 memory_used=true。
4. search_events 接收到的 memory 字段和 read_memory 输出一致。
```

### 3. Loading / Cache / Timings

```text
问题：
现在用户觉得慢，但不能只凭感觉判断是 intent parser、search_events 还是 LLM rewrite 慢。
如果没有 timings，大家会把时间花在错误瓶颈上。

原因：
当前 intent parser 默认是本地规则，理论上很快。
更可能慢的是远程 LLM rewrite、网络请求、同步执行和失败重试。

怎么改：
1. plan-day 每个阶段打点，写入 debug.timings_ms。
2. 优先缓存 LLM rewrite，因为它最可能耗时和消耗 token。
3. 再缓存 plan_result，但 key 必须包含 profile_hash、memory_hash、event_snapshot_hash。
4. cache 命中后仍检查活动是否过期、取消、不可用。
5. enable_llm_rewrite 改成配置项，超时后 fallback 到模板文案。
6. GET /api/agent/runs/{run_id} 返回 stage、progress、stage_message、cache_hit。

验收证据：
1. 第一次请求 debug.cache_hit=false。
2. 第二次相同请求 debug.cache_hit=true。
3. 第二次 total timings 明显下降。
4. debug 能看到 llm_rewrite_summary_and_reasons 的耗时。
5. LLM rewrite 超时或失败时，plan-day 仍能 completed，debug.used_fallback=true。
```

### 4. Intent / Rewrite

```text
问题：
intent parser 不应该为了 memory 把所有语义都变成硬约束。
“不要再推荐讲座”和“我更喜欢展览”表达的是不同强度。

原因：
硬约束会过滤候选，软偏好只影响排序。
如果把软偏好误判成硬约束，会导致候选过少或结果为空。

怎么改：
1. 当前 query 中明确排除的内容进入 hard_constraints 或 negative_keywords。
2. “更喜欢”“轻松一点”“离我近一点”进入 soft_constraints 或 style_tags。
3. “换一个”“不要重复刚才的”要转成 repeat_penalty 或 excluded_recent_event_ids。
4. rewrite prompt 只负责 summary/reason，不负责重新决定活动排序。
5. rewrite prompt 增加 prompt_version，进入 cache key。

验收证据：
1. 至少 5 条反馈/排除语义 query 有 eval case。
2. 每条 query 都有期望解析结果。
3. “不要太累”不会被解析成喜欢“太累”。
4. rewrite 输出稳定 JSON。
5. cache key 中能看到 prompt_version 或等价版本字段。
```

### 5. 前端展示

```text
问题：
用户等待时如果只看到 loading，会觉得系统卡住。
用户点反馈后如果没有确认文案，会不知道系统是否记住了。

原因：
第二阶段的产品目标不是只跑通接口，还要让用户理解 agent 正在做什么。

怎么改：
1. loading 页根据后端 stage_message 展示真实阶段。
2. cache_hit=true 时展示“命中缓存，正在返回结果”。
3. 结果页提供“喜欢”“不感兴趣”“查看来源”入口。
4. 点击反馈后给 toast 或状态文案，例如“已记录，下次会减少类似推荐”。
5. failed 状态保留 debug/rejection reason，开发模式可见。
6. 流式生成只做独立 demo，不阻塞主链路。

验收证据：
1. 录屏能看到 loading 阶段变化。
2. 录屏能看到点击“不感兴趣”并收到确认。
3. 前端请求体包含 event_id、plan_id、plan_item_id、run_id、feedback_type、feedback_source、metadata。
4. failed 页面能显示错误原因。
5. 流式探索有一句明确结论：可接入、暂不接入、或需要改用 WebSocket。
```
