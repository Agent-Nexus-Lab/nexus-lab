# 流式生成可行性调研

日期：2026-06-28

## 结论

本阶段正式主链路继续使用：

```text
POST /api/agent/plan-day
GET /api/agent/runs/{run_id}
```

也就是“发起 run -> 轮询状态 -> completed 后跳结果页”。前端已经支持 `stage`、`stage_message`、`progress`、`cache_hit`，足以支撑 Phase 2 结项演示。

流式生成不进入正式主链路，只保留为技术探索。小程序侧已预留 `pages/stream-demo/stream-demo`，用于验证 `wx.request({ enableChunked: true })` 和 `onChunkReceived` 是否能在当前基础库、开发者工具和真机上稳定接收分块数据。

## 当前判断

`enableChunked` 理论上可用于接收 chunked HTTP 响应，但它依赖微信基础库版本、真机环境和后端响应格式。它不适合作为本周强制交付项。

本周验收优先级：

1. loading 页展示真实阶段。
2. cache 命中时展示“命中缓存”。
3. failed 时展示错误原因和 debug 信息。
4. 结果页反馈能提交。
5. 流式只给出可行性结论。

## 方案对比

| 方案 | 前端接入 | 后端改动 | 稳定性 | 适合阶段 |
|---|---|---|---|---|
| 阶段式轮询 | 已完成 | 返回 `stage/progress/stage_message` | 高 | Phase 2 主线 |
| Chunked HTTP | 需要真机验证 | 新增 dev-only chunk endpoint | 中 | Phase 2 探索 |
| SSE | 小程序兼容性不确定 | 新增 SSE endpoint | 中 | Phase 3 备选 |
| WebSocket | 需要维护连接状态 | 新增 WS 服务 | 高 | Phase 3 正式流式 |

## 建议

Phase 2 使用阶段式轮询作为正式体验。它能解释 Agent 正在做什么，也能展示 cache 命中，工程风险最低。

Phase 3 如果要做真正的流式输出，建议优先评估 WebSocket。它比 chunked HTTP 更适合持续推送阶段、token、错误和最终结果，也更容易处理重连与状态同步。

## 小程序验证入口

开发页：

```text
pages/stream-demo/stream-demo
```

预期后端提供临时接口：

```text
POST /api/agent/stream-demo
```

响应可以先使用 JSON Lines：

```json
{"stage":"intent_parsing","message":"正在理解需求"}
{"stage":"search_events","message":"正在检索活动"}
{"stage":"completed","message":"验证完成"}
```

如果该页面在真机上无法稳定收到 chunk，则本阶段不继续投入流式开发，保留阶段式轮询方案。
