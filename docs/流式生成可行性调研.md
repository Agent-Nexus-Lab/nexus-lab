# 流式生成可行性调研

日期：2026-06-28

## 结论

本阶段正式主链路继续使用：

```text
POST /api/agent/plan-day
GET /api/agent/runs/{run_id}
```

也就是“发起 run -> 轮询状态 -> completed 后跳结果页”。前端已经支持 `stage`、`stage_message`、`progress`、`cache_hit`，足以支撑 Phase 2 结项演示。

流式生成已以“流式优先、轮询兜底”的方式接入正式生成入口。小程序侧保留 `pages/stream-demo/stream-demo` 作为诊断页，用于验证 `wx.request({ enableChunked: true })` 和 `onChunkReceived` 是否能在当前基础库、开发者工具和真机上稳定接收分块数据。

## 当前判断

`enableChunked` 理论上可用于接收 chunked HTTP 响应，但它依赖微信基础库版本、真机环境和后端响应格式。它不适合作为本周强制交付项。

本周验收优先级：

1. loading 页展示真实阶段。
2. cache 命中时展示“命中缓存”。
3. failed 时展示错误原因和 debug 信息。
4. 结果页反馈能提交。
5. 流式只给出可行性结论。

## 方案对比

| 方案 | 前端接入 | 后端改动 | 稳定性 | 适合阶段 |
|---|---|---|---|---|
| 阶段式轮询 | 已完成 | 返回 `stage/progress/stage_message` | 高 | 流式失败后的兜底主线 |
| Chunked HTTP | 已接入正式入口，仍需真机验证 | 新增 `/api/agent/stream-plan-day` | 中 | 流式优先路径 |
| SSE | 小程序兼容性不确定 | 新增 SSE endpoint | 中 | Phase 3 备选 |
| WebSocket | 需要维护连接状态 | 新增 WS 服务 | 高 | Phase 3 正式流式 |

## 建议

当前正式体验采用“Chunked HTTP 流式优先 + 阶段式轮询兜底”。流式可展示实时阶段，轮询兜底保证后端 endpoint 未就绪或真机 chunk 不稳定时仍可完成演示。

当前先用 chunked HTTP 接入正式入口；如果真机或线上网关对 `enableChunked` 支持不稳定，下一阶段建议评估 WebSocket。WebSocket 更适合持续推送阶段、token、错误和最终结果，也更容易处理重连与状态同步。

## 小程序验证入口

开发页：

```text
pages/stream-demo/stream-demo
```

预期后端提供临时接口：

```text
POST /api/agent/stream-plan-day
```

响应可以先使用 JSON Lines：

```json
{"stage":"intent_parsing","message":"正在理解需求"}
{"stage":"search_events","message":"正在检索活动"}
{"stage":"completed","message":"验证完成"}
```

如果正式入口在真机上无法稳定收到 chunk，则会自动回退到阶段式轮询方案，不影响生成主链路。
